# 面試 talking-points：20 分鐘怎麼講

> 用法：上面是「口袋小抄」（被問到就掃一眼）；中段是 3–4 個 **STAR 故事**（挑面試官有興趣的講）；
> 最後是「預期問題 + 一句話回答」。核心策略：**不要逐項報功能，講方法論 + 一兩個讓你被打臉的瞬間**——
> 那才顯出你「真的懂、而且會質疑自己」。

---

## 🎒 口袋小抄（30 秒電梯版）

> 「我是銀行 IT 出身，熟治理/合規/平台。我從零手刻了一個 GPT，把它走完整個 **LLM ＋ MLOps
> 生命週期**——資料工程、現代架構、訓練、部署治理，一直到**後訓練對齊（DPO/IPO/GRPO/PPO）**
> 和**關鍵演算法的數學推導**。模型只有 8M 參數、刻意小，**重點不是模型，是過程中的工程判斷
> 與方法論**：每個技巧先預測再實測、每個指標先講判準再驗證、每個假設量過才信。我最在意的一條
> 主線是『**質疑你的指標有沒有被汙染或被鑽**』——這正好是我稽核本行的直覺接到 ML 上。」

**一句話定位**：不是再多一個會調 API 的人，是「**用治理視角把 ML 工業化**」的 MLOps/平台人。

**關鍵數字**（被追問細節時拋）：
- 105 MB 中文維基實戰；MinHash+LSH 去重 11k 篇 **21.6 秒**
- 資料偵測器抓到 **21.6%** 文件殘留維基語法（聚合指標全說健康）
- DPO 兩種偏好軸 train-acc 都 100%，held-out **97% vs 9%**（死背 vs 真學）
- RLHF 無 KL 錨：RM 分數 3.7→**13.2**、輸出多樣性 100%→**6%**（reward hacking）
- 對齊四種都做：DPO / IPO / GRPO / PPO，各有對照實驗 + 數學推導
- 49 單元測試、CI 綠、`make verify` 驗收 playbook、uv 鎖依賴

## ⭐ STAR 故事（挑 3–4 個講）

### 故事 1 — reward hacking（最接稽核本行，先講這個）
- **S（情境）**：RLHF 用一個「學來的 reward model」當分數，讓 RL 去最大化它。
- **T（任務）**：我想親眼驗證一個我從稽核就熟悉的疑慮——「指標被當成目標去衝，會不會失真」。
- **A（行動）**：刻意拿掉 KL 錨、放 RL 用力優化 RM，同時**量兩把尺**：RM 分數（代理）與生成多樣性（真實）。
- **R（結果）**：RM 分數從 3.7 衝到 **13.2**（看起來超棒），但輸出 collapse 成「不管問什麼都吐同一串高分
  垃圾」、多樣性崩到 6%。那串垃圾落在 RM 訓練分布外、被誤判高分——policy 鑽了漏洞。**這就是 Goodhart's
  law，和稽核裡 KPI 造假同一回事**。防法＝KL 錨把 policy 綁在可信舊模型附近。
- **收尾金句**：「模型會過時，但『**會質疑自己那把尺**』這個紀律不會——這是我最想帶去團隊的東西。」

### 故事 2 — 資料品質：聚合指標的盲點（資料工程 + 稽核思維）
- **S**：把 1 MB 英文玩具換成 105 MB 中文維基後，熵 / 壓縮比 / 重複率全部顯示「資料健康 ✅」。
- **T**：我不信任「總分漂亮」，想知道資料裡到底有沒有髒東西。
- **A**：寫一套**資料品質偵測器**（7 個規則掃全語料、量化命中率、設門檻當 gate），心法是「看樣本發現
  問題 → 寫成一條偵測器 → 累積成資料的測試套件」。
- **R**：抓到 **21.6% 文件**殘留維基繁簡轉換語法 `-{zh-tw:..}-`——任何聚合指標都看不到。修一條清洗規則
  21.6%→0.05%，用監控面板 before/after 留視覺證據。
- **金句**：「這跟我疊稽核控制項是同一個思路——**一條一條把『看資料的眼睛』固化成程式**。」

### 故事 3 — 選對指標（方法論主線，展現嚴謹）
- **S**：做完 SFT（指令微調）後，我要評估它有沒有變好。
- **T**：用「維基 perplexity」量，結果顯示 SFT **變爛**了。
- **A**：我懷疑是**尺被汙染**——維基 perplexity 是預訓練的尺，被 alignment tax 拖低；連「對 gold 答案的
  loss」也被汙染（gold 是維基句、base 預訓練看過）。改量「**應答行為率**」（生成是否以定義句回應、免 gold）。
- **R**：行為率 base 29% → SFT **72%**，才看出 SFT 真價值。**同一顆模型、不同的尺、相反的結論。**
- **金句**：「評估難的不是算數字，是**選對該量的東西、並質疑它有沒有被汙染**。」

### 故事 4 — 從零到「能把式子推出來」（技術深度）
- **S**：我做了 DPO（直接偏好優化），但不滿足於「會調」。
- **T**：我想知道為什麼 DPO 不用 reward model、不用 RL 就能做對齊。
- **A**：親手把它從 RLHF 的約束最佳化推出來——解出最優 policy $\pi^*\propto\pi_\text{ref}e^{r/\beta}$、反解 reward、
  代進 Bradley-Terry，**那個算不出來的配分函數 $Z(x)$ 在相減時剛好抵消**。
- **R**：寫成 `derivations.md` + 一份「換你來推」的練習單；還用同一套數學解釋了實測的反直覺（β 越小漂移越大）。
- **金句**：「我要的是能**把式子推出來、把現象解釋清楚**，不是背 API。」

> 備用故事：**模型治理**（digest 身份 + registry + lineage + promotion gate，把 container/cosign 那套搬到模型）、
> **KV-cache 不一定更快**（GPU+小模型反而慢 18%，要量自己的 workload）。

## ❓ 預期問題 + 一句話回答

| 問題 | 回答 |
|---|---|
| 「為什麼不直接用 HuggingFace？」 | 用框架我只會「會用」；從零刻我才**看得懂內部、debug 得動、推得出式子**。這是深度的差異。 |
| 「8M 的玩具模型有什麼用？」 | 模型是載體，價值在**走完整條鏈的工程判斷**。小到能在筆電上幾分鐘跑完一個對照實驗——反而讓我能做**很多次** predict→實測→打臉的循環。 |
| 「你說的 MLOps 具體是什麼？」 | 可重現（uv lock）、可驗證（單元測試＋驗收 playbook＋CI）、可觀測（Prometheus/Grafana）、可治理（registry＋digest＋promotion gate＋lineage）、會腐壞要監控重訓（漂移＋重訓迴圈）。 |
| 「你跟一般 ML 工程師差在哪？」 | 我從**治理/稽核**進來：我的本能是「質疑指標、要可審計軌跡、上線要 gate」。reward hacking 那個實驗就是這個直覺的具體展現。 |
| 「下一步？」 | 規模化（雲 GPU）、把容器部上 k8s（我的本行）、或把這套寫成教材/書。 |
